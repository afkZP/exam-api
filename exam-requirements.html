ТРЕБОВАНИЯ К СДАЧЕ ЭКЗАМЕНА: Каталог личной библиотеки

Цель работы: Развернуть отказоустойчивую инфраструктуру для REST API, настроить процессы непрерывной интеграции (CI) и обеспечить безопасность хранения конфигурационных данных.

ВАЖНО: Проект должен быть полностью автоматизирован. Любые ручные манипуляции с базой данных или кодом после деплоя должны быть исключены.

ШАГ 1: ОРКЕСТРАЦИЯ (DOCKER COMPOSE)

Настройте совместный запуск приложения и базы данных, используя docker-compose.yml.

  1. Сервис БД: Добавьте сервис db. Используйте официальный образ (например, postgres:15-alpine).
  2. Связь сервисов: Настройте API-сервис так, чтобы он зависел от готовности базы данных (используйте healthcheck и depends_on с условием service_healthy).
  3. Конфигурация: Все параметры подключения (логин, пароль, имя БД) должны передаваться через переменные окружения, подгружаемые из файла .env.

ШАГ 2: УПРАВЛЕНИЕ РЕПОЗИТОРИЕМ И ВЕТКАМИ

Соблюдение методологии ветвления — критический аспект командной разработки.

  1. Публикация: Выложите исходный код проекта на GitHub.
  2. Структура веток:
    • main — стабильная версия (production).
    • dev — ветка для разработки и тестирования (staging).
  3. Branch Protection (Защита веток):

    Настройте правила для обеих веток в интерфейсе GitHub:

    • Запрет прямых пушей (Restrict pushes).
    • Обязательное создание Pull Request для внесения изменений.
    • Обязательное успешное прохождение всех CI-проверок перед мерджем.

ШАГ 3: НЕПРЕРЫВНАЯ ИНТЕГРАЦИЯ (CI)

Настройте автоматизацию проверок с помощью GitHub Actions.

Пайплайн должен запускаться автоматически при каждом push или Pull Request, направленном в ветки main или dev.

Этапы пайплайна:

  1. Линтинг: Проверка качества кода и соответствие стандартам (используйте ruff).
  2. Тестирование: Запуск Unit и Integration тестов (pytest).
  3. Сборка: Проверка собираемости Docker-образа (docker build).

ШАГ 4: БЕЗОПАСНОСТЬ

Защита чувствительных данных — приоритет №1.

  1. Исключение файлов: Настройте .gitignore. В репозитории не должно быть временных файлов, кэша Python и виртуальных окружений.
  2. Секреты: Файл .env с реальными паролями категорически запрещено коммитить в репозиторий. Он должен быть добавлен в .gitignore.
  3. Шаблон настроек: Для удобства других разработчиков создайте файл .env.example с перечислением необходимых переменных, но без секретных значений.

ИТОГОВЫЙ ЗАПУСК

Проверка работоспособности осуществляется командой: 

docker compose up -d --build
